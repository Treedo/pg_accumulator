# Реєстр подвійного запису: Ledger (Double-Entry Accumulation Register)

Цей документ описує архітектуру, концепцію та специфікацію нового типу регістру в pg_accumulator — **`ledger` (реєстр подвійного запису)**. Він розширює концепцію довільних аналітичних накопичувальних регістрів у природний механізм автоматично збалансованого бухгалтерського та фінансового обліку.

---

## 1. Проблема та рішення

У класичних регістрах накопичення (`balance`), таких як описані в [docs/README.md](docs/README.md#register_types), зміна ресурсів ведеться лінійно: наприклад, товар надійшов — `quantity` збільшилась, списався — `quantity` зменшилась.

Однак у фінансовому обліку кожна операція зачіпає щонайменше два рахунки одночасно (**подвійний запис**):
- **Дебет (Debit)** — збільшення активів або витрат.
- **Кредит (Credit)** — збільшення зобов'язань, власного капіталу або доходів.

Для розв'язання цієї проблеми вводиться тип регістру `kind := 'ledger'`. Він дозволяє вказувати в одному рядку і рахунок Дебету, і рахунок Кредиту разом із відповідними аналітичними вимірами (субконто). Такий підхід гарантує **природну фінансову консистентність** на рівні СУБД: гроші не матеріалізуються з нічого і не зникають безслідно.

## 2. Ключові концепції

### А. Однорядкове відображення операції
Замість двох окремих проводок (на дебет і кредит) розробник надсилає один-єдиний JSON-об'єкт:
```json
{
  "recorder": "invoice:2048",
  "period": "2026-05-30",
  "account_dr": "1010",
  "subconto_dr": {"cashbox_id": 1},
  "account_cr": "3610",
  "subconto_cr": {"customer_id": 412, "contract_id": 99},
  "amount": 2500.00
}
```
Це спрощує інтеграцію з боку прикладного програмного забезпечення (API/ORMs) та автоматично запобігає асиметрії транзакцій.

### Б. Динамічні субдані (Субконто) через JSONB
Традиційні регістри створюють статичні колонки вимірів. Для ledger-реєстру використовується динамічний підхід:
- **Глобальні виміри** (спільні для проводки), наприклад, `currency` (валюта), `organization` (організація). Задаються статичними колонками.
- **Субконто Дебету / Кредиту** (`subconto_dr` / `subconto_cr`) — довільний набір JSONB-параметрів. Їхнє унікальне поєднання автоматично згортається в `dim_hash` за допомогою криптографічно стійкої та швидкої функції хешування.

### В. Балансова тотожність та авто-знак рахунку
Кожен рахунок у системі класифікується за своїм типом:
- **Активний (Active):** сальдо збільшується за дебетом, зменшується за кредитом:
  $$\text{Balance} = \sum \text{amount\_dr} - \sum \text{amount\_cr}$$
- **Пасивний (Passive):** сальдо збільшується за кредитом, зменшується за дебетом:
  $$\text{Balance} = \sum \text{amount\_cr} - \sum \text{amount\_dr}$$
- **Активно-Пасивний (Active-Passive):** відображає як розгорнуті залишки дебету, так і кредиту для запобігання помилкового взаємозаліку (наприклад, аванси та заборгованість різних клієнтів на одному рахунку 3610).

## 3. Фізична та логічна архітектура

При створенні ledger-регістру `register_create(name := 'main_ledger', kind := 'ledger')` створюється наступна інфраструктура таблиць:

### 3.1. Таблиця рухів (`_movements`)
Параметризована глобальними вимірами та спільними ресурсами:
- `recorder text NOT NULL`
- `period timestamptz NOT NULL`
- `account_dr text NOT NULL`
- `subconto_dr jsonb`
- `account_cr text NOT NULL`
- `subconto_cr jsonb`
- `amount numeric(18,2) NOT NULL`
- (додаткові виміри, наприклад `currency text`)

### 3.2. Балансовий кеш (`_balance_cache`)
Це плоска таблиця, яка зберігає поточний стан окремо для кожного рахунку та комбінації його субконто:
- `account text NOT NULL`
- `dim_hash bigint NOT NULL` (хеш від глобальних вимірів + субконто даного рахунку)
- `subconto jsonb` (динамічні розрізи аналітики)
- `currency text` (якщо задано глобально)
- `amount_dr numeric(18,2) NOT NULL DEFAULT 0`
- `amount_cr numeric(18,2) NOT NULL DEFAULT 0`
- `PRIMARY KEY (account, dim_hash)`

### 3.3. Тригерна логіка (Схема віртуального розщеплення)
При `INSERT` одного руху в `_movements`, тригер `AFTER INSERT` автоматично виконує два оновлення типу `UPSERT` у `_balance_cache`:

1. **Для дебетової сторони:**
   ```sql
   INSERT INTO accum.main_ledger_balance_cache (account, dim_hash, subconto, amount_dr, amount_cr)
   VALUES (NEW.account_dr, _hash(NEW.subconto_dr, NEW.currency), NEW.subconto_dr, NEW.amount, 0)
   ON CONFLICT (account, dim_hash) DO UPDATE SET amount_dr = amount_dr + EXCLUDED.amount_dr;
   ```
2. **Для кредитової сторони:**
   ```sql
   INSERT INTO accum.main_ledger_balance_cache (account, dim_hash, subconto, amount_dr, amount_cr)
   VALUES (NEW.account_cr, _hash(NEW.subconto_cr, NEW.currency), NEW.subconto_cr, 0, NEW.amount)
   ON CONFLICT (account, dim_hash) DO UPDATE SET amount_cr = amount_cr + EXCLUDED.amount_cr;
   ```

Логіка видалення (`DELETE`) чи зміни (`UPDATE`) дзеркально віднімає обороти з відповідних чарунок кешу та таблиць періодичних підсумків (`totals_day`, `totals_month`, `totals_year`).

## 4. Забезпечення природної консистентності (Auditing & Soundness)

Для запобігання бухгалтерським помилкам розширення пропонує наступні механізми захисту:

### 4.1. Глобальні балансові рівняння
- **Загальна рівність дебетових і кредитових оборотів:**
  Сума всіх дебетів у кеші обов'язково повинна дорівнювати сумі кредитів:
  $$\sum \text{amount\_dr} \equiv \sum \text{amount\_cr}$$
- **Контроль закриття періоду (Locked Periods):**
  Будь-які записи до дати `period_close_date` автоматично викликають помилку, запобігаючи несанкціонованій зміні минулого фінансового результату.

### 4.2. Автоматична розгортка балансу
При отриманні балансу, функція `register_ledger_balance()` динамічно підставляє знаки балансу у відповідності до типу рахунку, який визначається за першими цифрами коду класу рахунку (наприклад, 1, 2, 9 — Активні; 4, 5, 7 — Пасивні, 3, 6 — Активно-пасивні).
